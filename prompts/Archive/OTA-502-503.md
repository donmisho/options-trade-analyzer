## OTA-502 — Scope
Earnings-in-window hard gate. Today the existing `EarningsInWindowGate` only catches earnings *near expiry*; it does not catch earnings *inside the trade window before expiry*. AMZN 260/270 May 15 (entry Apr 22, earnings Apr 29) scored EXECUTE despite earnings 7 days from entry.

Add — extend the existing `EarningsInWindowGate` (or add a sibling gate, whichever the existing class structure makes cleaner — confirm in Phase 1 below) so that:

- `earnings_in_window` = (earnings_date is not None) AND (entry_date ≤ earnings_date ≤ expiry_date)
- If `earnings_in_window` AND `days_to_earnings ≤ 7` → gate FAILs the candidate. Verdict becomes PASS at the verdict assignment site. Reason text: `"Earnings {mm-dd-yyyy} falls {N} trading days into trade window. Insufficient time to enter and exit before catalyst."`
- If `earnings_in_window` AND 8 ≤ `days_to_earnings` ≤ 13 → gate does NOT fail; instead the gate emits an `effective_DTE = days_to_earnings - 1` field and a `score_penalty = 15` (15-point scoring penalty). The scoring stage applies these.
- Otherwise (no earnings in window, or earnings ≥ 14 days from entry) → gate is a no-op; existing behavior preserved.

Wire the data path: read earnings date from the existing scan input (the candidate context already carries it for the existing gate). Do not add a new provider call.

The verdict assignment site (per OTA-503's description, `evaluation_routes.py` before the OTA-144 score band logic) must consume the gate's FAIL → PASS routing and surface the `reason` string in the response payload.

## OTA-502 — Acceptance criteria
- AMZN 260/270 May 15 regression case (entry Apr 22, earnings Apr 29) flips from EXECUTE to PASS solely on this gate (independent of OTA-503).
- A trade with earnings 10 days from entry receives `effective_DTE = 9` and a 15-point score penalty; verdict still goes through normal scoring on `effective_DTE` (does not auto-PASS).
- A trade with earnings 20 days from entry is unaffected.
- A trade with no earnings in window is unaffected.
- The reason string is present in the verdict payload when the gate FAILs (verifiable by curling `/evaluate/structured` against the AMZN regression case and inspecting the response).
- Unit tests cover all four branches (≤7 PASS, 8–13 penalty, no earnings in window, earnings outside window).
- The hard-gate ordering test still passes.

## OTA-503 — Scope
Negative EV hard gate. The existing `NegativeEVGate` is registered but the verdict assignment in `evaluation_routes.py` does not honor its FAIL — the OutcomeSummaryCard (built in OTA-294) shows a NEGATIVE EV badge, but the verdict still proceeds through the OTA-144 score band logic and emits EXECUTE.

Add the gate enforcement at the verdict assignment site. Modeled on OTA-141's credit % gate pattern:

```python
# In evaluation_routes.py (or wherever verdict is assigned),
# BEFORE the OTA-144 score band logic:

if expected_value < 0:
    verdict = PASS
    score = None  # or retain internally for diagnostic, suppress display
    reason = f"Negative expected value ({expected_value:.2f}). Trade fails primary quality gate."
    return verdict_response(verdict, score, reason)

# Existing OTA-144 score band logic continues only if expected_value >= 0
```

The NEGATIVE EV badge in OutcomeSummaryCard remains as the visual cue. This Story converts that cosmetic flag into an enforcing gate at verdict assignment.

If during Phase 1 you find that the existing `NegativeEVGate` registered in `app/analysis/hard_gates/` already runs at scan time and only the evaluation-route side is non-enforcing, the fix is solely at the evaluation-route side — do not duplicate the gate logic. If `NegativeEVGate` is not currently enforcing at scan time either, fix both paths.
