## Relevant Context — Do Not Deviate Without Escalation

Source: `insight_engine.md` §4 / §4.1 (the 7-phase pipeline, fixed order)
Rule: Fixed phase order: 1) Gates (RAW) → 2) Gates (DERIVED) → [adapter COMPUTED callback] → 3) Gates (COMPUTED) → 4) Scoring (weighted sum) → 5) Apply held gate penalties → 6) Adjustments → 7) Verdict band lookup. Within a phase, rules execute in junction `evaluation_order`. A score is never adjusted before it exists; a verdict is never assigned before adjustments apply (except via the halt-verdict path below). Gates are pure functions of inputs.

Source: `insight_engine.md` §3.6 / §4.1 (gate mechanics — config, not categories)
Rule: No hard/soft gate categories in code. `stop_if_fail = true` and fails → halt, record the terminal decision. `stop_if_fail = false` and fails → record, hold `score_penalty`, continue. Held penalties subtracted in Phase 5. Each Phase-6 adjustment clamps the score to `[0,100]` after it applies, and may force a floor/cap.

Source: OTA-695 confirmed assumption (one weighting source)
Rule: Scoring uses **exactly one** weighting source — the junction `weight`. Legacy `VerticalSpreadEngine`/`NakedOptionEngine` internal weight sets are superseded. No engine-internal weights.

Source: OTA-695 OD-2 (halt verdict from junction column)
Rule: When a `stop_if_fail = true` gate fails, the orchestrator inspects the junction row's `terminal_verdict`. If **non-null** → the candidate's verdict is that string, Phase-7 band lookup is **bypassed**, `terminal_phase` = the gate's phase, `final_score` null. If **null** → candidate halted with verdict null. `WAIT` (band 50–70) and `WAIT_FOR_EARNINGS` (halt verdict) remain distinct strings. `terminal_phase` is recorded wherever a candidate exits.
