## Relevant Context — Do Not Deviate Without Escalation

Source: `insight_engine.md` §6.3 (expression library — the closed set)
Rule: Condition expressions are drawn from a closed set the engine understands: comparison (`>=`, `<=`, `>`, `<`, `==`, `!=`); set membership (`IN`, `NOT IN`); range (`BETWEEN`); null semantics (`IS NULL`, `IS NOT NULL`); enum match (`EQUALS_ENUM`); named formula reference (`formula:<name>`). The library is the only evaluator; any expression form not in the set is a **load-time rejection**.

Source: `insight_engine.md` §6.3 + §2.3 (BETWEEN + atomicity)
Rule: `BETWEEN` is sugar — decomposed into **two atomic comparisons at load**, so the runtime never sees a compound form. There are **no AND/OR/NOT operators** anywhere in the evaluator; compound conditions are expressed as multiple atomic rules.

Source: `insight_engine.md` §6.3 (formula refs) + OTA-695 OD-1
Rule: `formula:<name>` resolves against the registered rule library and invokes the implementation with the candidate's named values and the rule's bound parameters: `(named_values, params)`. This Story owns resolution + invocation against the `FormulaRegistry` whose membership check OTA-699 already uses — extend the same registry object; do not fork a second one.
