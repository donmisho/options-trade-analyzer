### 2B — Persistence model implementation

Branch on Don's selection. Only implement one of the following.

**If Option A (DB canonical):**
- Add Alembic migration writing seed rows from current `.config.js` defaults into `strategy_configs`.
- Add API routes: `GET /api/v1/strategy-configs/{strategy_key}` (returns merged default + user override), `PUT /api/v1/strategy-configs/{strategy_key}` (writes user override).
- Frontend Strategy page reads via API on mount; writes via API on user edit; localStorage acts as read-through cache only (TTL or version-bust on app load).
- Delete redundant duplication in `.config.js` files; keep only structural defaults.
- Coordinate with Don: this adds rows to `strategy_configs` — record in OTA-523 Database Contract Actions tracker if not already there.

**If Option B (Frontend canonical):**
- `.config.js` files become the only source of truth.
- Frontend writes per-user overrides to localStorage only.
- Drop the `strategy_configs` table via Alembic migration.
- Add to OTA-523 Database Contract Actions tracker as a drop-table action.
- Delete any backend code that touches `StrategyConfig` model (model itself, any helpers).

**If Option C (Hybrid):**
- `.config.js` files = static defaults (read once at app build time, treated as constants).
- Add `strategy_configs` API routes: `GET /api/v1/strategy-configs/me` (returns user overrides only, empty if none), `PUT /api/v1/strategy-configs/me/{strategy_key}` (writes one strategy's overrides).
- Frontend resolution at runtime: `effective_config = {...defaults_from_config_js, ...user_overrides_from_api}`.
- No localStorage involvement.
- Strip any localStorage strategy-config code from the frontend.
- Coordinate with Don: this adds rows to `strategy_configs` only when users save overrides — record in OTA-523 if needed.
