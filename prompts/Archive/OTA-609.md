### Phase 2 — Documentation update

Add the Entra credential rotation procedure as a new ADR section to `auth-process.md`. The MCP Resource Server architecture itself was added by OTA-605; this Story adds only the rotation procedure.

The new ADR section follows the documentation governance rules:

- **State what IS** — no historical content, no alternatives considered, no mention of the bearer token approach that was tried and abandoned
- **Prescriptive rulebook** with Decision Date and Change Log only
- **Single bounded responsibility** — only the rotation procedure, not the broader MCP architecture (which lives in `mcp-server-spec.md`)

Suggested section structure:

```markdown
## ADR-N: MCP Entra Client Secret Rotation

**Decision Date:** <today>
**Change Log:** Initial decision (OTA-609)

The `ota-market-data` MCP server uses Microsoft Entra OAuth 2.1 with the
`ota-mcp-server` Entra app registration as the Authorization Server. The
client secret is stored in the `options-analyzer` Key Vault as
`mcp-entra-client-secret` and is also pasted into the claude.ai
connector's Advanced Settings during initial configuration.

### Rotation procedure

1. In the Entra portal, open the `ota-mcp-server` app registration → Certificates & secrets → New client secret. Set expiry per tenant policy (default 24 months).
2. Copy the secret **value** immediately — Entra masks it after the page is closed.
3. Update the `options-analyzer` Key Vault secret `mcp-entra-client-secret` with the new value:
   ```powershell
   az keyvault secret set --vault-name options-analyzer --name mcp-entra-client-secret --value "<new-secret>"
   ```
4. Update the claude.ai connector configuration (Settings → Connectors → `ota-market-data` → Edit → Advanced Settings → Client Secret) with the new value.
5. Reconnect the connector — claude.ai re-runs the OAuth dance with the new secret.
6. Verify connectivity by invoking any tool from a claude.ai conversation. Confirm `agent_run_log` records the call.
7. After verifying the new secret works, delete the old secret entry from the Entra portal app registration. Do not leave both active — overlapping secrets defeat the rotation purpose.

### Token validation reachability

The MCP Resource Server reads `mcp-entra-client-secret` from Key Vault on demand via `SecretsManager`. No App Service restart is required — the next request after the Key Vault update uses the new value. The Entra access tokens that claude.ai already holds remain valid until their natural expiry (~1 hour); claude.ai will silently refresh them using the new secret when needed.

### Per-environment rotation

Repeat the procedure for each environment if both dev and prod app registrations are being rotated. Today the same `ota-mcp-server` app registration serves both environments; if that ever changes (e.g., separate `ota-mcp-server-dev` and `ota-mcp-server-prod` registrations), rotate each independently.
```
