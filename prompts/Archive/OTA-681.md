Source: OTA-681 "Resolved design points" (these supersede any older plan phrasing ŌĆö anti-footguns)
1. Junction per-rule parameters = a SINGLE validated JSON blob: `parameters NVARCHAR(MAX)` with an
   `ISJSON` CHECK. NOT per-parameter columns and NOT a typed sub-table. (Per-blob validation vs.
   `engine_rules.parameter_schema` happens later at engine load ŌĆö not a DB constraint here.)
2. `engine_rules.rule_id` is an `INT IDENTITY` surrogate PK. Bronze (later) references the string
   `rule_key`, never the surrogate. Same pattern for `engine_strategies.strategy_id`.
3. App-scoped identity: every config table carries `owner_app_id` and a natural key `(owner_app_id, <key>)`
   as a UNIQUE constraint. `engine_apps` is the FK parent and holds the `SHARED` sentinel row.
4. Discriminator columns (`phase`, `tier`, `consumer_surface`, etc.) are free `varchar` ŌĆö NEVER `CHECK`
   enum constraints. Their valid domains live in `engine_lookups` (`SHARED`) and are validated at engine
   load, not by the database.

Source: `insight_engine.md` ┬¦ 6.6; OTA-681 description
`evaluation_order` uniqueness within `(strategy, rule.phase)` is enforced at engine load, NOT by a DB
constraint (phase lives on the rule, not the junction). Do not add a DB uniqueness constraint for it.
