## Phase 2 — Apply fix

**Change 1 — Widen exception logging (always apply, regardless of root cause):**

Modify the `except` block around `adapter.chat()` in `evaluation_routes.py` so it captures and logs:
- The full exception type and message
- If the exception is an `httpx.HTTPStatusError`, the response status code AND `response.text` (the Foundry error body)
- The trade ID, symbol, and structure type for correlation
- A truncated preview of the user_message that was sent (first 500 chars)

Use `logger.error(...)` (NOT print). Log before re-raising as `HTTPException(502, "AI evaluation failed")`.

**Change 2 — Fix the root cause** identified in Phase 1. Patch the specific code path. Examples by category:
- If naked option payload is missing a required field for prompt interpolation → defensive defaults in `_build_structured_user_message()` OR a pre-validation that raises a clear 422 instead of letting it reach Foundry
- If Foundry rejects on prompt size → trim the prompt for naked options, or restructure the SKILL.md so the naked branch uses a leaner template
- If Pydantic schema mismatch → fix the schema or the marshalling

Do not patch the symptom (catching and re-raising as success); patch the cause.

**Change 3 — Add a regression marker.** In whichever module owns the trade-evaluation regression set (likely `tests/` or a fixture file), add a comment or a fixture entry referencing OTA-558 and the symbol pair used for repro, so future maintainers know what this guards against.
