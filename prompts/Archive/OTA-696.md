## Relevant Context — Do Not Deviate Without Escalation

Source: `insight_engine.md` §2 principle 5 (Domain decoupling)
Rule: Engine machinery — config loader, rule evaluator, pipeline orchestrator, verdict assignment, result builder, persistence driver — contains **zero** references to any domain concept. Domain terms appear only inside (a) individual rule formula implementations registered with the engine, and (b) the consumer-specific input adapter. Enforced by an import-time check: the engine package must not import from any domain package.

Source: `insight_engine.md` §2 principle 2 (Strategy independence)
Rule: No strategy's logic may be expressed as a code branch. There is no `if strategy_id == X` / `if strategy_key == X` branch anywhere in engine code. Strategies are routed by config lookup only.

Source: `insight_engine.md` §8 (Boundaries)
Rule: The engine is NOT responsible for fetching market/domain data, computing domain values (Greeks, IV rank, ATR, SMAs), calling an LLM, or the persistence sink's mechanics (connection, credentials, transactions, retry, physical store). It therefore imports no domain package, no LLM client, and no DB driver.

Source: `insight_engine.md` §10 (Naming resolution)
Rule: The new framework package is the **Insight Engine** at `app/insight_engine/`. The pre-existing `app/agents/insight_engine.py` was renamed **Insight Communicator** — the new package does not collide with it. Confirm the package path is clear during Phase 0.

Source: OTA-695 (parent feature) — confirmed assumptions
Rule: Engine core is **mechanism only** — no options-domain terms, no rule values. Values live in the `engine_*` tables (seeded under OTA-680); domain math lives in the adapters/rule libraries (Wave 3). This Story builds the package boundary and its enforcement, nothing domain-specific.

## Scope
Create the engine package skeleton and the machinery that enforces its domain boundary. This Story does **not** implement engine behavior — each capability is its own downstream Story (see Out of scope).

### Phase 0 — Discovery (READ ONLY, no edits)
1. Confirm `app/insight_engine/` does not yet exist, or report exactly what precursor work (OTA-709 / OTA-710 / OTA-711 / OTA-689) already placed there. Do not overwrite precursor files.
2. Enumerate the domain package roots the guard must forbid: `app/analysis/`, `app/api/`, `app/models/`, `app/agents/`.
3. Identify the concrete module path(s) for the LLM client (Azure AI Foundry / anthropic client) and the DB drivers in use elsewhere (e.g. `pyodbc`, `sqlalchemy`, `azure.*`), so the forbidden-import list is accurate rather than guessed.
4. Identify the test directory layout and how CI discovers/runs tests, so the new scan test is wired in correctly.

Report findings in your final summary. If Phase 0 reveals that any shared file must change, or that `app/insight_engine/` already holds conflicting content, **HARD STOP and report** before editing.

### Phase 1 — Implementation
1. Create the `app/insight_engine/` package with `__init__.py`.
2. Define the public API **surface** in `__init__.py`:
   - `evaluate(...)` — the single public entry point, present as a **stub that raises `NotImplementedError`** with a docstring stating it is implemented in OTA-701. Document the intended call shape (candidate stream, `strategy_key`, required `source_app_id`, injected adapter + sink) without implementing it.
   - Establish the `__all__` export pattern. Export `evaluate` now. Leave a clearly-marked extension point for the dataclass exports OTA-697 will add. Do **not** create or export dataclasses — that is OTA-697.
3. Implement the domain-decoupling guard:
   - A guard module (e.g. `app/insight_engine/_guard.py`) that statically scans every `.py` under `app/insight_engine/` (AST parse; collect all `import` / `from … import` statements) and raises a dedicated exception (e.g. `EngineDomainLeakError`) if any import references a forbidden root: the domain packages from Phase 0, any LLM client module, or any DB driver.
   - The guard also fails on any `if strategy_id ==` / `if strategy_key ==` branch anywhere in the package.
   - Invoke the guard at **package import time** (from `__init__.py`) so a domain leak fails loudly on import, not silently at runtime. Reuse the same scan for the CI test below — one source of truth, two invocation points.
4. Add the CI test:
   - A test (in the Phase 0 test location) that runs the same AST/scan over `app/insight_engine/` and fails the build on any forbidden import or on any `strategy_id ==` / `strategy_key ==` branch.

## Acceptance criteria
- `app/insight_engine/` exists with `__init__.py` and a defined public API surface: the `evaluate(...)` entry-point stub plus the established export pattern for the OTA-697 dataclasses.
- An import-time check raises a dedicated exception on any import from a domain package, an LLM client, or a DB driver.
- A CI test AST-scans the package and fails the build on a forbidden import.
- No `if strategy_id == …` / `if strategy_key == …` branch exists anywhere in the package (grep-verified by the test).
- No dataclass bodies, loader, validator, expression library, pipeline, callback, result builder, bronze contract, or sink are implemented.

## Out of scope
- Engine behavior of any kind: dataclasses (OTA-697), config loader (OTA-698), startup validation (OTA-699), expression library (OTA-700), pipeline orchestrator (OTA-701), COMPUTED callback (OTA-702), result-record builder (OTA-703), bronze record contract (OTA-704), persistence sink interface (OTA-705), `source_app_id` enforcement (OTA-706), integration tests (OTA-707), bronze DDL/migration (OTA-708).
- Any options-domain term, rule value, or strategy logic.
- Wiring the package into `app/main.py` or any application call site.
- Any edit to a shared file (`app/main.py`, `app/database.py`, `web/src/api/client.js`).

## Verification steps
1. `python -c "import app.insight_engine"` succeeds on a clean tree (guard passes when no leak is present).
2. Temporarily add a forbidden import (e.g. `from app.analysis import x`) inside a package module; confirm the import-time guard raises `EngineDomainLeakError`; remove it.
3. Run the test suite; confirm the scan test passes on the clean package and fails on the injected-leak fixture.
4. `grep -rnE "strategy_(id|key)\s*==" app/insight_engine/` returns nothing.
5. Confirm no import of any domain package, LLM client, or DB driver exists anywhere in the package.
