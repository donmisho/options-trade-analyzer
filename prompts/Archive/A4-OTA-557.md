## Relevant Context — Do Not Deviate Without Escalation

**Source: business-rules.md → Position Lifecycle (canonical)**
A position is "expired" when its expiration date has passed. Auto-archive on the Positions page sweep promotes expired positions to the archive automatically when the page loads, so the user does not see stale rows. Auth tier on Positions page load is `require_read`. Archive operations require `require_write`. This auth-tier asymmetry is suspect #1 for the symptom in this Bug.

**Source: ticket OTA-557 → Diagnostic logs already in place**
Three console lines fire on every Positions page load (added in OTA-556):
- `[PositionsPage] N active position(s) have no expiration in trade_structure` — `console.warn` (keep permanently)
- `[PositionsPage] Auto-archiving N expired position(s)` — `console.info` (remove once root cause confirmed)
- `[PositionsPage] Auto-archive failed for SYMBOL (id)` — `console.warn` (keep permanently)

**Source: CLAUDE.md → Date format (canonical)**
Date comparisons in browser use `formatDate()` and the backing date string format. If the bug is browser-specific date parsing, fix the parsing — do not patch with locale strings.
