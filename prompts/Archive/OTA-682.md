## Relevant Context — Do Not Deviate Without Escalation

Source: `insight_engine.md` § 6.2; OTA-682 description
The importer is a **one-time, build-time** mechanism. It reads `Scoring Parameters.xlsx`, conceptually
restructures it into Rules / Strategies / Junction / Lookups, and writes the `engine_*` tables. After
import the workbook is historical — **the engine never reads the workbook at runtime** (the engine's
config loader, a later feature, reads tables only). No engine code path may import or read the workbook.

Source: OTA-682 description
This story builds the **loading mechanism**, not the final content. The correctness of *what* gets seeded
(rule decompositions, divergence resolutions, formula definitions, per-strategy parameters) is owned by the
content stories (OTA-683, 685, 686, 687, 688, 690). Because the importer is idempotent and re-runnable, it
is re-run after those content stories land. Build it to faithfully load whatever the workbook currently
contains; do not bake content decisions into the importer.

Source: OTA-682 acceptance criteria
- Populates `engine_rules`, `engine_strategies`, `engine_strategy_rule_junction`, and the `engine_lookups`
  sets, under `owner_app_id = 'OTA'` (or `'SHARED'` for genuinely generic enumerations).
- `engine_apps` rows already exist (structural seed from OTA-681); the importer does NOT recreate them, it
  references them for FK resolution.
- **Idempotent.** A clean re-run against a fresh DB produces identical rows. A re-run against a populated DB
  is a controlled UPSERT keyed on the natural keys — never duplicate rows:
  `(owner_app_id, rule_key)`, `(owner_app_id, strategy_key)`, `(strategy_id, rule_id)`,
  `(owner_app_id, lookup_set, lookup_key)`.
- Every JSON blob written (`parameters`, `parameter_schema`, `referenced_named_values`,
  `compatible_structures`, `verdict_band_set`, `payload`) is validated as well-formed JSON BEFORE insert.
- Standalone script, NOT part of the engine runtime.

Source: `CLAUDE.md` House Style + `architecture-plan.md` § 3
- Use the project's standard DB access for the write path; do not invent a second connection pattern.
- If the importer authenticates to Azure SQL via Entra (managed identity / `DefaultAzureCredential`), use
  the async `.aio` credential variants per the project's async-credential rule. (As a standalone script
  this is lower-risk than an async request handler, but stay consistent with the codebase.)

Source: OTA-682 description
Document, in the script's module docstring / README note, that **production edits live in the tables
(admin UI), not in the spreadsheet**; re-import is for rebuilds only.

## Scope
Build a standalone, idempotent seed importer that:
1. Reads `Scoring Parameters.xlsx` (inspect the real sheet/column layout first).
2. Maps workbook content → `engine_rules`, `engine_strategies`, `engine_strategy_rule_junction`,
   `engine_lookups` (FK-resolving against the existing `engine_apps` rows).
3. Validates every JSON blob as well-formed before insert.
4. Performs natural-key UPSERTs (insert-or-update, never duplicate) so the script is safely re-runnable.
5. Carries the documented note that the workbook is build-time only and runtime edits happen in-table.

## Acceptance criteria
- Running the importer against a DB that has OTA-681's tables populates the four content tables; all FK
  references resolve against the seeded `engine_apps` rows.
- Idempotent: clean re-run on a fresh DB → identical rows; re-run on a populated DB → controlled upsert on
  the natural keys, zero duplicate rows.
- Every JSON value written is validated as well-formed JSON before insert (invalid JSON aborts that row
  with a clear error, not a silent partial write).
- The importer is a standalone script; no engine/runtime code path reads the workbook.
- The build-time-only / runtime-edits-in-table note is present in the script and/or its README.
