## Relevant Context — Do Not Deviate Without Escalation

**Source: build-execution.md § Part 4 — Deployment Recording**
Rule: Every successful deployment to a non-local environment must produce a deploy-log record. Required fields at minimum: build ID (CI run ID), environment (`dev` / `prod`), deployment timestamp (UTC ISO 8601), commit SHA, ticket keys parsed from commit message. This Story implements the OTA-side fulfillment of that profile-level rule.

**Source: architecture-plan.md § Schema Migration Strategy**
Rule: Schema changes ship via Alembic migrations. Additive-only this Story (new table, new index). No drops, no column renames. Migration must be reversible (downgrade defined).

**Source: architecture-plan.md § Data Isolation Invariant**
Rule: Every CRUD endpoint that takes a resource ID must filter by `user_id`. **Exception applicable here:** the `deploy_log` table is observability data, not user-scoped data. Records have no `user_id` column. Both the POST (deploy-token-authed, system-write) and the GET (BFF-session-authed, read-only) operate on the table as a whole. Document this exception in code comments so future audits don't flag it as a violation.

**Source: architecture-plan.md § AI Adapter Contract / general routes pattern**
Rule: New routes register under `app/<domain>/<domain>_routes.py` and are wired into `app/main.py` via the standard `include_router(...)` call. Do not put new routes in `app/main.py` directly.

**Source: CLAUDE.md § House Style Rules**
- Date format: `mm-dd-yyyy hh:mm` via `formatDate()` from `web/src/utils/formatDate.js`. No locale strings.

- `var(--bg2)` (#161b22) is restricted to filter bars, QuoteBar, and pill badge backgrounds only — never on table rows, headers, or expansion panels.
- Buttons sized to content, never full-width unless explicitly designed that way.
- Environment badges should follow the existing badge-color convention; teal for "prod" matches the requested style (per the Story description).

**Source: CLAUDE.md § Commit Message Convention**

## Scope

### Backend
1. **Schema.** New table `deploy_log` per the spec in the Jira description: columns `id` (PK), `build_id`, `environment`, `deployed_at`, `commit_sha`, `ticket_keys`, `notes`, `created_at`. Index on `deployed_at desc`. Alembic migration is additive-only with a defined downgrade. Model declared in the appropriate place (`app/database.py` if that's where existing models live; confirm in Phase 1).

2. **Routes.** New route file `app/changelog/changelog_routes.py` (or the OTA-idiomatic equivalent confirmed in Phase 1). Two endpoints:

    - `POST /api/v1/changelog/record`
        - Auth: header `X-Deploy-Token` matched against the Key Vault secret `deploy-recorder-token`. Mismatch → 401. Missing → 401.
        - Body schema (Pydantic): `{ build_id: str, environment: Literal["dev", "prod"], commit_sha: str, ticket_keys: list[str] = [], notes: Optional[str] = None }`
        - Validators: `commit_sha` exactly 40 hex chars; `build_id` non-empty; each entry in `ticket_keys` matches `^OTA-\d+$`.
        - Response: 201 with `{ id: int }`. Standard 400 on validation failure with structured error.
        - Observability: log every write at INFO level with the build_id and environment.

    - `GET /api/v1/changelog?limit=50&environment=prod`
        - Auth: standard BFF session cookie. No deploy token here.
        - Query params: `limit` default 50, max 200; `environment` optional filter (`dev` or `prod`).
        - Response: reverse-chronological list of full row records.
        - No `user_id` filter — observability data exception documented in the Data Isolation Invariant context above.

    Wire the new router into `app/main.py` via `include_router(...)`. Do not place route definitions in `app/main.py` directly.

3. **Ticket-key parser.** If a parser already exists for the auto-transition automation, reuse it. If not, implement as a small utility: regex `OTA-\d+` over the commit message, dedupe, return list. Handle edge cases: no tickets in message → empty list; lowercase `ota-123` → not matched (the convention is uppercase only).

### Frontend
1. **Route.** Add `/changelog` to the router config (likely `web/src/App.jsx` or `web/src/router.jsx` — confirm in Phase 1).

2. **Page component.** New `web/src/ChangeLogPage.jsx` (or under `web/src/pages/` if that's the convention; confirm in Phase 1) rendering:
    - Heading: "Change Log"
    - Table with columns: Deployed At · Environment · Build ID · Commit · Tickets · Notes
    - Deployed At: `formatDate(row.deployed_at, { withTime: true })` → `mm-dd-yyyy hh:mm` per house style
    - Environment: badge — `dev` neutral, `prod` teal. Use existing badge component if one exists; otherwise build inline using token CSS vars.
    - Build ID: link `https://github.com/donmisho/options-trade-analyzer/actions/runs/{build_id}` (target `_blank`, `rel="noopener noreferrer"`); link text is the run ID itself.
    - Commit: link `https://github.com/donmisho/options-trade-analyzer/commit/{commit_sha}`; link text is 7-char short SHA.
    - Tickets: each entry in `ticket_keys` rendered as a link to `https://tmtctech-team.atlassian.net/browse/{key}`; comma-separated.
    - Notes: text or empty cell.
    - Empty state: "No deploys recorded yet." centered.
    - Loading state: spinner or skeleton consistent with other pages.
    - Mobile breakpoint: table either stays horizontally scrollable or collapses to card layout. Match existing pages' approach.

3. **API client.** Add `getChangeLog({ limit, environment })` to `web/src/api/client.js`. POST is a server-only call (workflow YAML), so no client method for it.

4. **Nav.** Add "Change Log" link to `Layout.jsx` placed below "Settings" as a peer item (not a sub-item). Match the existing nav-link styling.

### Workflow YAML
Only modify if the Pre-work secrets are confirmed present. Otherwise skip and report blocked-on-Don.

1. **`.github/workflows/deploy-to-dev.yml`.** Add a final step (after the deploy-success confirmation) that POSTs to `https://<dev-host>/api/v1/changelog/record` with header `X-Deploy-Token: ${{ secrets.DEPLOY_RECORDER_TOKEN }}` and body `{ build_id: "${{ github.run_id }}", environment: "dev", commit_sha: "${{ github.sha }}", ticket_keys: <parsed from commit message>, notes: null }`. Use the same ticket-key parser regex (or call the backend helper if exposed via a util script — Phase 1 confirms which is cleaner).

2. **`.github/workflows/swap-staging-to-prod.yml`.** Add the equivalent final step posting to the prod host with `environment: "prod"`. The intermediate `deploy-to-prod.yml` (which only deploys to the staging slot, not the user-facing prod) does NOT post.

3. **Failed-deploy handling.** Both YAML steps must be conditional on prior-step success. A failed deploy must not produce a row.

## Acceptance criteria
1. After a successful dev deploy, a row appears in `deploy_log` with the GitHub Actions run ID, `environment="dev"`, deploy timestamp, commit SHA, and parsed ticket keys.
2. After a successful prod swap, a row appears with `environment="prod"`.
3. The `/changelog` page loads and displays the last 50 rows in reverse-chronological order.
4. Build ID, Commit, and each Ticket link resolve correctly when clicked.
5. The "Change Log" link appears below "Settings" in the left nav as a peer item.
6. Failed deploys do not create rows (verified by inducing a deploy failure in dev and confirming no row appears).
7. The POST endpoint requires a valid `X-Deploy-Token` header — calls without it return 401; calls with a wrong token return 401.
8. The page renders under 1 second for the default 50-row limit on dev.
9. House style applied throughout: `formatDate()`, no `$` prefix, dark theme tokens only, no inline hex, `var(--bg2)` not used on table rows or headers.
10. The page works on mobile breakpoint (table either horizontally scrollable or collapses to cards).
