## Relevant Context — Do Not Deviate Without Escalation

**Source: architecture-plan.md § AI Adapter Contract (canonical)**
The unified contract:
```
chat(system: str, user: str, max_tokens: int) -> dict
```
Returns: `{text, input_tokens, output_tokens, model, provider}`. Both adapter implementations must implement this exact dict shape. Any caller can invoke `chat()` and not know which provider is behind it.

**Source: ticket OTA-536 → Two stacks today**
- `app/ai/` — httpx-based, JSON-structured, used by `evaluation_routes.py`. **KEEP.**
- `app/providers/ai/` — SDK-based, prose-shaped, used by `agent_routes.py` (or its renamed equivalent post-OTA-541). **DELETE.**

**Source: ticket OTA-536 → Recommended cure**
Keep the httpx/JSON stack (`FoundryEvalAdapter`). Migrate `agent_routes.py` to use it via `chat()`. Then delete `app/providers/ai/` entirely.

**Source: CLAUDE.md → Async credential rule**
All async Azure SDK calls in async handlers must use `azure.identity.aio` async variants. Any credential code touched in this migration uses the async variant.

**Source: CLAUDE.md → Observability**
Both call paths must log to `agent_run_log` (or the canonical AI observability table) with the same field shape — `provider`, `model`, `input_tokens`, `output_tokens`, etc.

## Phase 2 — Apply migration

### Formalize the ABC

Create or update `app/ai/base.py`:

```python
from abc import ABC, abstractmethod
from typing import TypedDict


class ChatResult(TypedDict):
    text: str
    input_tokens: int
    output_tokens: int
    model: str
    provider: str


class AIAdapter(ABC):
    """Canonical AI invocation contract per architecture-plan.md § AI Adapter Contract."""

    @abstractmethod
    async def chat(self, system: str, user: str, max_tokens: int) -> ChatResult:
        ...
```
