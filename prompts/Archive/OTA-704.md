## Relevant Context — Do Not Deviate Without Escalation

Source: `insight_engine.md` §4.3 (engine owns the bronze contract)
Rule: The engine **owns the shape** of what gets persisted and **drives the write**, but does not own write mechanics. Map each `ResultRecord` into two logical streams, stamping `source_app_id`, `config_version`, `evaluated_at` on **every** record:
- `CandidateSnapshot` — one per candidate, every run (full named-value set + result-record summary; enables later re-evaluation, which is why non-stopping failures are retained).
- `EvaluationDecision` — one per rule evaluation (rule_key, phase, `evaluation_order`, tier, value, params, pass/fail, `stop_if_fail`, `was_terminal`, score contribution, decision reason).
Promote-to-column vs. keep-in-`payload_json` split follows the §4.3 **golden rule**: anything used in a WHERE/JOIN/GROUP BY is a promoted column; everything else lives in `payload_json`. This keeps the bronze zone uniform across apps.

Source: `insight_engine-schema-ddl.md` §3.1 (physical landing is the sink's concern)
Rule: The OTA sink lands both logical streams into **one** physical table (`bronze_evaluations`, discriminated by `record_type`) + the `bronze_payload_contract` registry. That collapse is the **sink's** job (Wave 4), **not** engine core. Engine core emits the **two logical streams** and must reference **no** table name and **no** DB.
