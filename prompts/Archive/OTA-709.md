## Relevant Context — Do Not Deviate Without Escalation

**Source: `insight_engine-schema-ddl.md` §1 (Conventions)**
- All engine tables live in `dbo`. Framework tables are name-prefixed: `engine_*` for configuration.
- **No domain literals as constraints.** Discriminator columns (`phase`, `candidate_type`, `verdict`, `record_type`) are free `varchar`. Their valid domains are registered as `SHARED` rows in `engine_lookups` and validated at engine load, never expressed as a `CHECK` enum.
- Junction mechanical fields (`evaluation_order`, `stop_if_fail`, `score_penalty`, `weight`, `enabled`, `rationale`) are typed columns. `terminal_verdict` joins this set as a typed column — **not** a JSON field.

**Source: `insight_engine.md` §3.4 (Strategy and the strategy×rule junction)**
- The junction is the single home of per-strategy rule configuration. Adding `terminal_verdict` here keeps that invariant: the column is per (strategy, rule) binding, not per rule.
- "There are no defaults inherited from the rule. There are no overrides of a base rule. The junction row is the only place the strategy's configuration for the rule exists."

**Source: `insight_engine.md` §3.8 (Verdict bands), reinforced by OTA-695 (Engine core extraction)**
- The engine treats verdict labels as opaque strings. `EXECUTE`, `WAIT_FOR_EARNINGS`, `A`, `PASS` — all opaque to the engine.
- **Mechanism this column enables:** when a `stop_if_fail = true` gate fails and its junction row carries a non-null `terminal_verdict`, the engine's orchestrator (built under OTA-701 — Pipeline orchestrator with gate mechanics + halt-verdict path) sets the candidate's verdict from this column and bypasses the Phase-7 band lookup. Null (the default for most rows) means "halted, no special verdict label — score and verdict columns stay null in bronze."

**Source: `architecture-plan.md` § 2 (Migration Discipline)**
- Schema changes follow **expand/contract**. This Story is the **expand** half: the column is added nullable so existing code paths and rows are unaffected. No contract step is anticipated (nothing drops).
- New migrations land in `alembic/versions/`. The baseline is `f9e59a180957`. In dev/staging, `init_db()` runs `alembic upgrade head` at startup. In production, migrations are applied manually as part of the deploy procedure.
- The sibling Story OTA-681 (Define and create the engine_* runtime configuration tables (DDL + Alembic)) shipped the original `engine_*` DDL via Alembic. The migration produced by this Story must follow the same Alembic pattern OTA-681 established — **read that migration file before writing this one.**

**Source: `insight_engine.md` §6.6 (Startup validation)**
- Domain validation of `terminal_verdict` values against the verdict-domain `engine_lookups` set is the responsibility of OTA-699 (Startup validation suite (§6.6 + evaluation_order uniqueness)) at engine load, **not this Story**. This Story adds the column with **no CHECK constraint, no FK, no default value other than NULL.** Any DB-level enforcement is wrong here.
