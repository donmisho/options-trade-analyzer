## Relevant Context — Do Not Deviate Without Escalation

Source: `insight_engine.md` §1 (engine is domain-agnostic) + OTA-707 ticket
Rule: Build a fixture rule library + a **synthetic** input adapter (e.g. "evaluate apples by sweetness and price") + an in-memory `ConfigSource` + an in-memory sink, and run the engine end-to-end. These tests prove the engine has **zero** options-specific behavior — the core is exercised entirely on synthetic config, with **no options imports anywhere in the test path**.

Source: OTA-707 ticket — required tests
Rule: Five tests:
1. **End-to-end pass** — candidates flow through all 7 phases to a verdict, with a full trace and bronze records emitted.
2. **Swap** — re-run with a different adapter, strategy, and `source_app_id`; get a different but correctly-structured result, with bronze records stamped with the new app id (proves domain-agnosticism + `source_app_id` stamping).
3. **Non-stopping gate** — a `stop_if_fail = false` failure retains the candidate through scoring and emits both the failed-gate decision and the final verdict.
4. **Halt-verdict path** — a stopping gate carrying a `terminal_verdict` halts the candidate and emits that verdict, bypassing the band lookup.
5. **evaluation_order-collision rejection** — a config with two enabled rules sharing an order within a `(strategy, phase)` is rejected **at load** (the OTA-683 regression guard).

Source: OTA-695 OD-1 (dual formula sources in sync)
Rule: Seed **both** formula sources in sync per test — the SHARED `engine_lookups` `('SHARED','formula_registry')` set **and** the live registered fixture rule library — so the OTA-699 dual-membership + drift checks pass for valid fixtures (and can be deliberately desynced to prove the drift error).
