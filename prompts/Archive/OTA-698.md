## Relevant Context — Do Not Deviate Without Escalation

Source: `insight_engine.md` §6.1 / §6.2 (tables are the source of truth)
Rule: The runtime model is built from the `engine_*` tables — **as-built, five tables**, not "three": `engine_apps` (scope), `engine_rules`, `engine_strategies`, `engine_strategy_rule_junction`, `engine_lookups`. The loader reads **tables only** — never `Scoring Parameters.xlsx` (the workbook is a one-time seed handled in OTA-682). Only `enabled = 1` rows participate.

Source: `insight_engine.md` §6.5 (reload behavior)
Rule: Restart-only. The loader is the single place the runtime model is built; nothing downstream re-reads the tables mid-run.

Source: OTA-696 guard + OTA-705 (sink injection) — **boundary resolution (the flag raised at OTA-696)**
Rule: The engine package must not import any DB driver (`pyodbc`, `sqlalchemy`, `azure.*`, etc.). The loader therefore consumes an **injected config source** — define a `ConfigSource` interface (analogous to `PersistenceSink`) that yields raw rows per table; the loader transforms rows → in-memory model. The concrete Azure-SQL-backed `ConfigSource` is **app-side wiring (Wave 4)**, out of scope here. Provide an in-memory/dict-backed `ConfigSource` for tests (OTA-707 will use it).

Source: `insight_engine.md` §3.4 / §4 (RuleSet resolution + ordering)
Rule: Each strategy resolves to a `RuleSet` whose bound rules are ordered by `evaluation_order` **within phase**; gate phases sequenced RAW → DERIVED → COMPUTED. `RuleSet` is the resolved runtime view defined in OTA-697.

Source: `insight_engine-schema-ddl.md` §1–§2 (JSON columns + scope)
Rule: Scope every load by `owner_app_id` (`OTA` content + `SHARED` library). Parse JSON columns into typed structures: `parameters`, `parameter_schema`, `referenced_named_values`, `verdict_band_set`, `compatible_structures`, and lookup `payload`.

## Scope
Build the config loader that hydrates the in-memory model from an injected `ConfigSource` and resolves each strategy into a `RuleSet`.

### Phase 0 — Discovery (READ ONLY, no edits)
1. Read the OTA-697 dataclasses (`Rule`, `Strategy`, `JunctionRow`, `RuleSet`, `NamedValue`) and the `__init__.py` export surface.
2. Read the OTA-696 guard's forbidden-import list to confirm the loader must stay DB-free.
3. Use `insight_engine-schema-ddl.md` §2 column names as the row-shape reference.
4. Check whether any config-source / loader abstraction already exists from precursors. Report findings. If a DB-backed loader already exists in-package, **HARD STOP and report** (it would violate the guard).

### Phase 1 — Implementation
1. Define a `ConfigSource` interface (protocol/ABC) with read methods returning raw rows for each of the five `engine_*` tables (e.g. `fetch_apps`, `fetch_rules`, `fetch_strategies`, `fetch_junction`, `fetch_lookups`). No DB code in the interface or the loader.
2. Provide an in-memory `ConfigSource` (dict/list-backed) for tests.
3. Implement the loader: consume the `ConfigSource`, filter to `enabled = 1`, parse JSON columns into typed structures, and build the in-memory model scoped by `owner_app_id`.
4. Resolve each `Strategy` into a `RuleSet`: pair each rule with its `JunctionRow`, ordered by `evaluation_order` within phase, gate phases sequenced RAW → DERIVED → COMPUTED.
5. Compute and capture a deterministic `config_version` hash over the loaded configuration, for later stamping onto result records.

## Acceptance criteria
- Loader hydrates all five `engine_*` tables (incl. `engine_lookups`) into the in-memory model via the injected `ConfigSource`, scoped by `owner_app_id`.
- Each strategy resolves to a `RuleSet` ordered by `evaluation_order` within phase; gate phases RAW → DERIVED → COMPUTED.
- JSON columns are parsed into typed structures.
- No code path reads `Scoring Parameters.xlsx`; no DB driver is imported anywhere in the package.
- A deterministic `config_version` hash is produced.
- An in-memory `ConfigSource` exists for tests; the OTA-696 guard and CI scan still pass.
