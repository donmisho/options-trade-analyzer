## Relevant Context — Do Not Deviate Without Escalation

Source: `insight_engine.md` §1 / §4.3 (cross-app provenance)
Rule: Every `engine.evaluate(...)` call carries a `source_app_id` (`OTA`, `FFL`, `STK`, …). The engine validates it is present and non-empty and stamps it through to **every** emitted record — the result record and both bronze streams. This is what keeps a shared cross-app bronze zone queryable per-app. There is **no** default/fallback `source_app_id` in engine code.

Source: scope clarification (Wave boundaries)
Rule: This Story enforces the **parameter contract** inside the engine. Wiring production OTA call sites (`evaluation_routes.py`, etc.) to call the engine is **Wave 4 app-side** and out of scope here; the "OTA call sites pass `OTA`" criterion is satisfied by the integration tests (OTA-707) and any existing in-engine call path — **do not** edit domain/app route files in this Story.

## Scope
Make `source_app_id` a required, validated parameter of `evaluate(...)` and verify it is stamped through every emitted record.

### Phase 0 — Discovery (READ ONLY, no edits)
1. Read the OTA-701 `evaluate(...)` signature and the OTA-703 / OTA-704 stamping paths.

### Phase 1 — Implementation
1. Require `source_app_id` on `evaluate(...)`; fail fast with a clear error on missing/empty.
2. Ensure it is stamped onto the result record and every `CandidateSnapshot` / `EvaluationDecision`.
3. Remove any default/fallback `source_app_id` in engine code.
4. Ensure in-engine call paths (tests/fixtures) pass `"OTA"` where representing OTA.
