## Relevant Context — Do Not Deviate Without Escalation

**Source: `insight_engine.md` §6.3 (Expression library) and §6.6 (Startup validation)**
- The engine's expression library is a closed set. `formula:<name>` references a registered implementation, looked up at evaluation time.
- At engine load, OTA-699 will validate that every `formula:<name>` reference in `engine_rules.formula_ref` resolves to a registered implementation. A missing reference is a load failure.

**Source: OTA-695 (Engine core extraction) — dual-validation contract**
- The membership check runs against **two independent sources at load**:
  1. A **table contract** — the SHARED `engine_lookups` set `('SHARED', 'formula_registry')`, this Story's second deliverable. Answers "is this name in the contract."
  2. A **live registered rule library** — the in-memory registry populated by the screening rule library (arrives with a later Wave-3 feature, not yet ticketed). Answers "does an implementation exist."
- Both checks fail the load independently, with distinct error messages.
- The two sources must agree at the end of load. Drift between them is itself a defect class OTA-699 will catch as a third distinct error.
- For pre-Wave-3 development the live-registry check operates against a stub registry; integration test fixtures (covered under OTA-707) will seed both sources in sync.

**Source: `insight_engine-schema-ddl.md` §2 (engine_lookups)**
- `engine_lookups` rows are keyed on `(owner_app_id, lookup_set, lookup_key)` with a JSON `payload`.
- This Story writes rows under `owner_app_id = 'SHARED'`, `lookup_set = 'formula_registry'`, one row per formula name (`lookup_key`). The SHARED scope is the cross-application rule-library scope — formulas live here because they can be used by any consuming app.

**Source: OTA-682 (the seed importer)**
- The importer is idempotent on its upsert paths. `engine_lookups` upserts on `(owner_app_id, lookup_set, lookup_key)`.
- For this Story to land cleanly, the importer must write the new lookup set on a clean re-run. Confirm in Phase 0 that the importer's `engine_lookups` ingestion supports the `SHARED` scope and the new set.

## Scope

Two artifacts ship as one commit; they must agree row-for-row.

### Artifact 1 — The doc

A complete, deduplicated list of every `formula:<name>` reference used in `engine_rules.formula_ref`, with per-formula notes:

- **Name** — the exact string used after the `formula:` prefix.
- **Intent** — what the formula computes, in plain language.
- **Signature** — input named values (with type if it varies — `decimal`, `int`, `bool`, `str`) and output type. Use the same input naming the input catalog (`§5.1` / `business-rules.md`) uses.
- **Notes (optional)** — proxy status if applicable (e.g. "current implementation is ATM IV / 0.60 used as IV-rank proxy; planned replacement is true IV-percentile from adapter"), or any other implementation caveat.

Format: a markdown table or a per-formula section, whichever matches the existing convention in the OTA repo's claude_context docs. Phase 0 confirms the convention. Default if no convention exists yet: a new file `claude_context/engine-formula-registry.md` with a section per formula and a Change Log header following the conventions used in `insight_engine-schema-ddl.md`.

The list must be:
- **Complete** — every distinct `formula_ref` value in `engine_rules` is represented.
- **Deduplicated** — one entry per name, even if multiple rules reference it.
- **Inclusive of OTA-686's additions** — every formula introduced under that Story is present.

### Artifact 2 — The SHARED `engine_lookups` seed rows

For every entry in Artifact 1, one row in the source structure the importer reads to populate `engine_lookups`. Each row:

- `owner_app_id = 'SHARED'`
- `lookup_set = 'formula_registry'`
- `lookup_key = <formula name>` (e.g. `'theta_gamma_ratio'`)
- `payload` JSON shaped to carry the same intent / signature notes from the doc. Suggested shape (Phase 0 confirms or adjusts to match existing-lookup convention):
  ```json
  {
    "intent": "...",
    "inputs": ["theta", "gamma"],
    "input_types": {"theta": "decimal", "gamma": "decimal"},
    "output_type": "decimal",
    "notes": "..."
  }
  ```
  If existing SHARED `engine_lookups` rows use a different payload convention, **match that convention** — do not invent a competing shape.

The seed must be importer-loadable: re-running the importer against a clean DB results in exactly these rows under `('SHARED', 'formula_registry', <name>)`.

### Drift discipline

The doc and the lookup payloads are two views of one contract. At every point in the work:

- If the doc lists a formula, the lookup set has a row for it.
- If the lookup set has a row, the doc has an entry for it.
- The intent / signature notes match across both.

A single source of truth for the content is ideal — Phase 0 explores whether the doc can be generated from the lookup-source-data, or vice versa, or whether they're maintained in parallel with a manual cross-check. Either is acceptable; Don approves the approach before Phase 1.
