## Relevant Context ‚Äî Do Not Deviate Without Escalation

Source: `insight_engine-schema-ddl.md` ¬ß3 (bronze persistence)
Rule: Create the two bronze tables **exactly** per ¬ß3, in **one** Alembic migration:
- `bronze_evaluations` ‚Äî single append-only landing table holding both engine record streams, discriminated by `record_type` (`'SNAPSHOT'` | `'DECISION'`). **No foreign keys**; all references out are denormalized soft keys. Promoted columns per the ¬ß4.3 filter rule (identity/provenance: `record_id` PK, `record_type`, `run_id`, `snapshot_id`, `source_app_id`, `config_version`, `engine_version`, `evaluated_at`, `payload_version`; SNAPSHOT-only: `candidate_type`, `strategy_key`, `symbol`, `user_id`, `subject_type`, `subject_id`, `final_score`, `verdict`, `terminal_phase`; DECISION-only: `rule_key`, `phase`, `tier`, `evaluation_order`, `passed`, `stop_if_fail`, `was_terminal`, `score_contribution`); everything else in `payload_json` with an `ISJSON` check. Indexes per ¬ß3.
- `bronze_payload_contract` ‚Äî self-describing payload registry, unique on `(record_type, discriminator_value, payload_version)`.
- `record_type` CHECK (`'SNAPSHOT'`,`'DECISION'`) and `payload_json` `ISJSON` check present.

Source: OTA-708 ticket (sequencing / scope)
Rule: **Additive only** ‚Äî two net-new `bronze_*` tables; **no existing table altered**. Not a core-build blocker (engine core is tested against the in-memory sink). It is the prerequisite for the concrete OTA sink (Wave 4). Downgrade cleanly drops both tables. Safe against the shared `options-analyzer-db`.

## Scope
Author one Alembic migration creating the two bronze tables exactly per ¬ß3.

### Phase 0 ‚Äî Discovery (READ ONLY, no edits)

### Phase 1 ‚Äî Implementation
1. Create one revision off the current head implementing the ¬ß3 DDL for `bronze_evaluations` and `bronze_payload_contract` (columns, PKs, CHECK constraints, indexes ‚Äî including the filtered indexes ‚Äî and `ISJSON` checks).

3. Do not alter any existing table.

## Acceptance criteria
- Alembic upgrade creates `bronze_evaluations` and `bronze_payload_contract` in `dbo` with exactly the columns, PKs, CHECK constraints, and indexes in `insight_engine-schema-ddl.md` ¬ß3.
- No foreign keys on `bronze_evaluations`; soft keys only.
- `record_type` CHECK (`'SNAPSHOT'`,`'DECISION'`) and `payload_json` `ISJSON` check present.
- Migration is additive; `downgrade` cleanly drops both tables.
- Single Alembic head after the revision (no branch).
