## Relevant Context — Do Not Deviate Without Escalation

**Source: OTA-682 (the importer being extended)**
- The importer is a standalone one-time script. The engine never reads `Scoring Parameters.xlsx` at runtime; production edits happen in the tables.
- The importer is **idempotent** on its upsert paths, each keyed on the natural key for its table. The junction's natural key is `(strategy_id, rule_id)`.
- A clean re-run against an empty DB produces identical rows. A re-run against a populated DB upserts existing rows to match the source — never duplicates.
- Every JSON blob written (`parameters`, `parameter_schema`, `referenced_named_values`, `compatible_structures`, `verdict_band_set`, `payload`) is validated as well-formed JSON before insert. `terminal_verdict` is **plain `varchar(32)`, not JSON** — different validation path.

**Source: `insight_engine-schema-ddl.md` §2 (junction definition)**
- `terminal_verdict` is a nullable typed column on `engine_strategy_rule_junction`, joining `evaluation_order`, `stop_if_fail`, `score_penalty`, `weight`, `enabled`, `rationale` as a mechanical field.
- The column is `varchar(32)` — written values must be a string of length ≤ 32, or null.
- The DB has no CHECK constraint on the value. Runtime domain validation against `engine_lookups` is OTA-699's job at engine load, **not the importer's**.

**Source: `insight_engine.md` §6.2 (build-time seed vs. runtime source of truth)**
- After this importer runs, the tables — not the workbook — are authoritative. The importer is run for rebuilds, not as an ongoing edit path.
- The source-of-record decisions for which junction rows carry which `terminal_verdict` are owned by OTA-711 (Configure terminal_verdict per junction row + verdict-domain seed). This Story (OTA-710) writes whatever the source-of-record specifies; it does not decide values.

## Scope

Extend the existing seed importer so its junction-row upsert path handles the `terminal_verdict` column:

1. **Source ingestion.** Wherever the importer parses junction-row inputs from `Scoring Parameters.xlsx` (or whatever source structure the importer reads), add ingestion of a `terminal_verdict` field. If the source row's `terminal_verdict` is absent, blank, or whitespace-only, the importer treats it as null. If present and non-blank, the importer reads it as a string.
2. **Type validation (importer-side only).** Before insert/upsert, validate each non-null value: it must be a non-empty string of length ≤ 32 (the column type). If a value violates this, the importer fails the row with a structured error message naming the offending `(strategy_key, rule_key)` pair, the offending value, and the violation. No silent truncation. No silent fallback to null.
3. **Upsert path.** Include `terminal_verdict` in the `(strategy_id, rule_id)`-keyed upsert. On clean insert it's part of the new row. On update it's set to the source value (which may explicitly be null — the importer must distinguish "absent in source" from "null in source" only if your source format supports that distinction; otherwise absent and null both write null).
4. **No domain validation.** The importer does **not** check `terminal_verdict` values against any `engine_lookups` set. That check happens at engine load (OTA-699's responsibility). The importer's only validation is type + length.
5. **No code changes outside the importer.** The junction model class (if there is one in the importer's local definitions) gets the column added. Application-side ORM models, engine code, API routes — untouched.
