## Relevant Context — Do Not Deviate Without Escalation

**Source: mcp-server-spec.md → ADR-1 (Mount on Existing App Service)**
Rule: The MCP server lives in `app/api/mcp_routes.py` mounted at `/mcp` inside the existing FastAPI process. No new App Service, no new domain, no separate deploy. Conforms to Pattern 7 in architecture-plan.md.

**Source: mcp-server-spec.md → ADR-2 (Microsoft Entra as OAuth 2.1 Authorization Server)**
Rule: Auth follows the OAuth 2.1 Resource Server pattern from the MCP spec. Entra is the AS; OTA's `/mcp` is the RS. The MCP Python SDK's `mcp.server.auth.TokenVerifier` validates each access token: JWKS signature, audience matches `api://<ota-mcp-server-app-id>`, scope contains `mcp.invoke`, unexpired. The server publishes `/.well-known/oauth-protected-resource/mcp` per RFC 9728. The BFF cookie/CSRF middleware does **not** apply to `/mcp/*`.

**Source: mcp-server-spec.md → ADR-3 (User Binding via Entra OID Claim)**
Rule: Every authenticated request resolves to a `User` row by extracting the `oid` claim from the validated JWT and looking up the row where the Entra OID column matches. Mirrors the BFF resolver in `app/auth/dependencies.py` / `app/auth/session_manager.py`. The exact column name is to be discovered in Phase 1.

**Source: auth-process.md → Files Reference**
Rule: `app/auth/dependencies.py` is "FastAPI auth dependencies (cookie-first; JWT fallback during migration)". The "JWT fallback" path is a legacy MSAL bridge, scheduled for removal — **do not reuse it**. The MCP path is a separate, pure JWT Resource Server pattern that validates against Entra JWKS, not local sessions.

**Source: auth-process.md → Identity Credential Management**
Rule: Existing BFF credentials (`entra-bff-cert`, `session-encryption-key`) are scoped to user identity via the BFF flow. The MCP path adds a **new** credential `mcp-entra-client-secret` in the same `options-analyzer` Key Vault. The MCP path does not touch BFF credentials.

**Source: architecture-plan.md → Pattern 3 (Two-Track Observability)**
Rule: Every AI/tool invocation writes one OTel span and one row to `agent_run_log`. Failed observability writes are fire-and-forget — they must never block the primary tool response. The morning's `mcp_tool_observability` wrapper in `mcp_routes.py` is preserved and updated to use the resolved `user_id` from the Entra OID claim.

**Source: architecture-plan.md → Pattern 1 (Provider Adapter Layer)**
Rule: This Story does not call providers directly — it only swaps the auth layer. The existing `get_smas` tool from OTA-607 keeps its existing `_get_provider("market_data")` path.

2. In `app/api/mcp_routes.py`:
   - **Remove** the bearer-token middleware (the `verify_mcp_bearer` dependency from the morning's commit)
   - **Remove** the admin-role system principal resolver (`get_system_principal`) — the new model resolves user per-request from JWT claims, not from a cached singleton
   - **Remove** the import of the bearer Key Vault secret name
   - **Add** an Entra `TokenVerifier` instance configured with:
     - JWKS URL: `https://login.microsoftonline.com/{settings.ENTRA_TENANT_ID}/discovery/v2.0/keys`
     - Expected issuer: `https://login.microsoftonline.com/{settings.ENTRA_TENANT_ID}/v2.0`
     - Expected audience: `settings.ENTRA_MCP_APPLICATION_ID_URI`
     - Required scope: `settings.ENTRA_MCP_REQUIRED_SCOPE`
   - **Wire** the `TokenVerifier` into the `FastMCP` instance per the MCP SDK's documented Resource Server integration pattern (consult the SDK shape captured in Phase 1)
   - **Add** an OID-to-User resolver:
     ```python
     async def resolve_user_from_token(claims: dict, session: AsyncSession) -> User:
         """Map a validated Entra JWT's oid claim to a User row.

         Mirrors the BFF resolver. Returns the User instance; raises
         HTTPException(403) if no User row matches the oid claim.
         """
         oid = claims.get("oid")
         if not oid:
             raise HTTPException(403, "Token missing oid claim")
         result = await session.execute(
             select(User).where(User.<entra_oid_column> == oid)
         )
         user = result.scalar_one_or_none()
         if user is None:
             raise HTTPException(403, f"oid {oid} not provisioned in users table")
         return user
     ```
     (Replace `<entra_oid_column>` with the column name discovered in Phase 1.)
   - **Update** `mcp_tool_observability` to take the resolved `user_id` as an argument rather than reading from a cached system principal. Each tool function passes the resolved user.
   - **Verify** the existing `get_smas` tool body (from OTA-607) is unchanged structurally — only the argument it passes to `mcp_tool_observability` shifts from the system principal's id to the per-request resolved user id.

3. Add the OAuth Protected Resource Metadata discovery endpoint:
   - Register `GET /.well-known/oauth-protected-resource/mcp` (at the FastAPI app level, not inside the MCP mount, so it's reachable without auth)

     ```json
     {
       "resource": "https://oa.tmtctech.ai/mcp",

       "scopes_supported": ["mcp.invoke"],
       "bearer_methods_supported": ["header"]
     }
     ```
     The `resource` value reflects the deployment URL; for local dev it's `https://127.0.0.1:8000/mcp`.

- `mcp-entra-client-secret` is read from `options-analyzer` Key Vault via `SecretsManager`.

### User binding

- The OID-to-User resolver looks up the `User` row by the Entra OID column discovered in Phase 1.
- A token with valid signature, scope, and audience but an `oid` claim that does not match any `User` row returns 403 with a clear message.
- A token missing the `oid` claim returns 403.

- `GET /.well-known/oauth-protected-resource/mcp` returns valid RFC 9728 JSON listing Entra as the authorization server and `mcp.invoke` as the supported scope.
- The endpoint is reachable without auth.
- The 401 response on `/mcp/*` includes the `WWW-Authenticate: Bearer resource_metadata="..."` header pointing at this discovery URL.
