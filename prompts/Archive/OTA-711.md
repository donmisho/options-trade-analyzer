## Relevant Context — Do Not Deviate Without Escalation

**Source: `insight_engine.md` §3.4 (Strategy and the strategy×rule junction)**
- Per-strategy rule configuration lives in junction rows only. There are no defaults inherited from the rule, no overrides of a base rule.
- A `terminal_verdict` value on a junction row applies only to that (strategy, rule) binding. Setting `WAIT_FOR_EARNINGS` on the earnings gate for one strategy does not affect any other strategy's earnings-gate binding.

**Source: `insight_engine.md` §3.8 (Verdict bands) + OTA-695 (Engine core extraction)**
- Verdict labels are opaque strings to the engine. `WAIT` (a band verdict, score 50–70) and `WAIT_FOR_EARNINGS` (a halt verdict from the junction column) are **distinct strings**. They must not be conflated.
- A halt with non-null `terminal_verdict` bypasses the Phase-7 band lookup. A halt with null `terminal_verdict` leaves the candidate's verdict null.

**Source: OTA-683 (Decompose compound rules into atomic rules (during seed))**
- The earnings gate is decomposed into a 4-route tree of atomic rules. Each route has its own `stop_if_fail` and ordering. **Only the stopping routes** (those with `stop_if_fail = true`) can meaningfully carry a `terminal_verdict` — a non-stopping rule's failure records and continues, so its verdict can only come from Phase-7 anyway. Setting `terminal_verdict` on a non-stopping junction row is a configuration error and must not be done here.

**Source: `insight_engine.md` §6.6 + OTA-699 (Startup validation suite)**
- At engine load, OTA-699 will validate every non-null `terminal_verdict` value against the verdict-domain `engine_lookups` set for the strategy's `consumer_surface`. **Every halt verdict string this Story introduces must also exist as an `engine_lookups` row**, or the engine will refuse to load.
- The verdict-domain lookup set already carries `EXECUTE`, `WAIT`, `PASS` (the screening band verdicts seeded under OTA-682). Halt verdicts are **additions, not replacements** — do not remove or alter the existing rows.

**Source: OTA-682 (the seed importer)**
- Production edits go through the engine tables (admin UI when one exists), not by re-importing. Re-import is for rebuilds, and is what this Story uses to load the new data into the dev DB.
- `engine_lookups` writes follow the same idempotent upsert pattern: keyed on `(owner_app_id, lookup_set, lookup_key)`.

## Scope

Three artifacts ship as one commit:

### Artifact 1 — Source-data decisions for `terminal_verdict`

In `Scoring Parameters.xlsx` (or whatever the canonical source structure is — confirmed in Phase 0), add a `terminal_verdict` value to every junction row that should carry one. Defaults to null (or blank) for the vast majority.

**Known cases (must be set):**
- **Earnings gate × SP** stopping route(s) → `WAIT_FOR_EARNINGS`
- **Earnings gate × WG** stopping route(s) → `WAIT_FOR_EARNINGS`
- **Earnings gate × TR** stopping route(s) → `WAIT_FOR_EARNINGS`

The exact rule_keys correspond to OTA-683's earnings-gate decomposition. Phase 0 identifies the rule_keys; this Story sets the column on the matching junction rows. If TR's earnings-gate binding is non-stopping (`stop_if_fail = false`), it gets no `terminal_verdict` (per the §3.6 rule above).

**LT (long-dated):** explicitly **not** set. LT's 7-day buffer past expiry (OTA-687) means earnings within the trade window is generally informational, not a hard halt. If discovery shows LT does carry a stopping earnings rule, surface that question to Don before setting a value.

**Other halting gates:** for each `stop_if_fail = true` junction row that is **not** an earnings-gate row, decide per gate whether a distinct halt verdict adds value. The default is null. Examples to consider in Phase 0:
- Data-completeness gates (e.g. ATR_14 missing, IV_rank missing) — could emit `WAIT_DATA_INCOMPLETE` or similar, or stay null.
- Liquidity gates — likely stay null (a halt with no special label is fine).

Phase 0 enumerates every `stop_if_fail = true` junction row and proposes a value (or null) for each. Don approves the proposal before Phase 1.

### Artifact 2 — Verdict-domain `engine_lookups` seed rows

For every distinct non-null `terminal_verdict` value introduced in Artifact 1, add an `engine_lookups` row to the verdict-domain set for `consumer_surface = SCREENING`. Phase 0 confirms the exact `(owner_app_id, lookup_set)` keys the existing band verdicts use (likely `('OTA', 'screening_verdicts')` or similar — discovered, not assumed).

Each new row's `payload` JSON should follow the same shape as the existing band-verdict rows (intent / label / classification). At minimum:

```json
{
  "label": "Wait for Earnings",
  "kind": "HALT_VERDICT",
  "description": "Halt emitted by the earnings gate when an earnings announcement falls inside the trade window."
}
```

The exact shape must match what the existing rows use. Match the convention, don't invent a new one.

### Artifact 3 — Documentation

Update the strategy / junction documentation source(s) wherever junction configuration is documented (typically alongside how OTA-687's strategy config is documented; Phase 0 identifies the exact location). For each non-null `terminal_verdict` set in Artifact 1, the docs should explicitly call out:
- Which (strategy, rule) bindings carry which halt verdict, and why
- That `WAIT` (band verdict) and `WAIT_FOR_EARNINGS` (halt verdict) are distinct strings

Format and location follow whatever convention OTA-687's strategy config docs established. Do not invent a new doc file unless one obviously belongs and no existing doc fits.
