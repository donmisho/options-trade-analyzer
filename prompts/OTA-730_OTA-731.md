## Relevant Context — Do Not Deviate Without Escalation

**Source: insight_engine.md §3.6 — gates are configuration, not categories**
A gate's failure behavior is fully described by junction fields: `stop_if_fail` (true = halt; false = record + continue), `score_penalty` (reduction if it continues), and `evaluation_order`. "Hard gate" / "soft gate" are not separate phases — they are the same gate phase with different junction settings. Gate mechanics come from junction rows, **not** code.

**Source: OTA-695 OD-2 (terminal_verdict, resolved) — for the earnings gate**
A stopping gate whose junction row carries a non-null `terminal_verdict` (e.g. `WAIT_FOR_EARNINGS`) makes that string the candidate's verdict and **bypasses the Phase-7 band lookup**. `WAIT_FOR_EARNINGS` is a halt verdict, not a fourth score band. OTA-699 validates the `terminal_verdict` against the verdict-domain lookup. The formula does not set the verdict — the junction does; the formula just evaluates the gate condition.

**Source: migration plan (S4.5) — earnings gate (OTA-730)**
Decompose the 4-route earnings tree into **four atomic rule formulas** (atoms seeded under OTA-680). Each is a hard gate or a post-scoring adjustment depending on the route's effect; behavior (stop/penalty/order/terminal_verdict) is junction-driven.

**Source: migration plan (S4.6) + audit §5b #5, §7.5 #2 — NegativeEVGate (OTA-731)**
Move `negative_ev_gate.py` into the rule library as a registered formula and **consolidate** it with the duplicate filter path at `vertical_engine.py:265` so there is a single negative-EV rule, not two parallel ones.
