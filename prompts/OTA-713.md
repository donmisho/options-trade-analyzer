## Relevant Context — Do Not Deviate Without Escalation

**Source: insight_engine.md §5 (input adapter contract)**
The adapter is the **only** place domain knowledge lives on the input side of the engine. The adapter:
- fetches raw data from the consumer's source (Schwab),
- computes DERIVED values,
- produces COMPUTED values on demand via callback,
- delivers candidates with named values populated to the requested tier,
- declares an **input catalog** (name, tier, type, null semantics, producer reference) for every value it produces.
The adapter **does not** run rules, assign scores, or know about strategies, junctions, or verdict bands. Engine code that references a domain name is a defect. This boundary is enforced architecturally.

**Source: insight_engine.md §5.2 (COMPUTED callback contract) — signature fixed by OTA-702 (Code & Test Complete)**
```
adapter.populate_computed(candidates: list[Candidate], needed: set[str]) -> list[Candidate]
```
The engine passes the surviving candidate set plus the set of COMPUTED names any remaining active rule needs; the adapter returns those candidates with the requested fields populated. **The engine owns this signature — match it exactly. Do not redefine it.** OTA-713 only conforms the adapter method to this signature; the COMPUTED math itself arrives in OTA-716.

**Source: insight_engine.md §5.1 (input catalog)**
The adapter publishes a catalog of every named value it produces: `name`, `tier` (RAW / DERIVED / COMPUTED), `type`, `null semantics`, `producer reference`. The engine validates at startup that every named value referenced by an active rule is present in the catalog with a compatible type. In OTA-713 the catalog may be sparse (producers land in OTA-714–722, full publish in OTA-723), but the **structure** must conform to §5.1.

**Source: insight_engine-migration-plan.md (bucket #2 — input adapters)**
> All adapters can share lower-level providers (Schwab client, Black-Scholes implementation, etc.) but each is its own boundary.
The Schwab client and the Black-Scholes implementation are pulled out as shared providers under `app/ota_adapters/_shared/` so the position-health and directional adapters can reuse them rather than duplicate them.

**Source: CLAUDE.md — House Style / Provider routing**
- Never hardcode a provider name. Always route market data through `_get_provider()` or settings. The Schwab client provider in `_shared/` must preserve this — it wraps the existing provider path, it does not introduce a new hardcoded Schwab reference.
- Never hardcode prompts in Python; load via `skill_loader.py`. (Not expected to apply here, but holds if any path touches skills.)

**Source: CLAUDE.md — Infrastructure lessons (async credential rule)**
If any code in `_shared/` touches Azure (e.g., Key Vault for Schwab tokens) inside an async FastAPI path, it must use the `.aio` async credential variants (`azure.identity.aio`), never the synchronous ones, and must not instantiate `DefaultAzureCredential()` at module level (lazy-init, double-checked locking). Synchronous Azure calls block the event loop and only manifest as 90+ second hangs in production.

**Source: OTA-695 boundary (engine core — Code & Test Complete; do NOT modify)**
The engine package `app/insight_engine/` is built and its stories are Code & Test Complete. This prompt imports the engine's **public** contract types (e.g. the `Candidate` type) but must not modify anything under `app/insight_engine/`. If the engine's public surface seems to require a change, STOP and report — do not edit engine core.

## Scope

This is the **skeleton + interface + shared-provider** story for the options-chain adapter. It creates the package boundaries and the contract surface; it does **not** move any producer logic (that is OTA-714–722).

3. **Wire the package imports** so the engine can import `OptionsChainAdapter`, instantiate it, and call all three methods without error.
