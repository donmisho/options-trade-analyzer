## Relevant Context — Do Not Deviate Without Escalation

**Score breakdown table — exact column set and component order.** Source: QQQ v2 sample:

```
### Score breakdown

| Component | Score | Weight | Contribution |
|---|---|---|---|
| Expected value (EV) | 78 | 0.20 | 15.6 |
| Structure fit (vs profile) | 12 | 0.20 | 2.4 |
| Cushion / strike placement | 41 | 0.15 | 6.2 |
| Technical alignment (SMAs) | 65 | 0.15 | 9.8 |
| IV environment | 58 | 0.10 | 5.8 |
| DTE fit | 80 | 0.10 | 8.0 |
| Liquidity (bid-ask, volume) | 91 | 0.10 | 9.1 |
| **Total** | | | **56.9 → 57** |
```

Component order is **significant**. It matches the sample row-for-row and the audit grouping that the QA skill expects. Do not re-sort.

**Formatting overrides for the table — confirm with the QA skill.**
- Component score column: integer (e.g. `78`, `12`, `91`) — sample uses no decimals
- Weight column: two decimals (e.g. `0.20`, `0.15`, `0.10`)
- Contribution column: one decimal (e.g. `15.6`, `2.4`, `9.8`)
- Total row: bold; format `<raw_one_decimal> → <adjusted_integer>` when a penalty applied, else just the integer total

This sample-specific formatting deviates from the general `##.00` score format in `CLAUDE.md` House Style Rules. The deviation is intentional — the breakdown table prioritizes readability over uniform precision. Document the override in a code comment at the Score breakdown builder. **Confirm with Don before final** that this integer/one-decimal mixed precision is the desired contract, since it's a sample-specific override of the canonical `##.00` rule.

**Weights — source of truth.** Weights live in `business-rules.md` Strategy Scoring section. **Do not hardcode** the weights in the assembler. Read them from the configuration the scoring engine uses. If the configuration is not directly accessible from the assembler module, expose it via a small helper in the scoring service that the assembler imports.

**Penalty adjustment rule.** When the raw weighted sum is adjusted (e.g., cushion penalty applied), the Total row renders both values: `<raw> → <adjusted>`. Source: QQQ v2 sample Total row `**56.9 → 57**` — the raw 56.9 was rounded to 57 (no penalty in this sample; the arrow shows the raw-to-displayed transform).

When a real penalty fires (e.g., cushion penalty subtracts 5 from a 78.9 raw → 73.9 adjusted), render `**78.9 → 73 (cushion penalty)**` with the penalty reason in parentheses.

**No `$` prefix in the score breakdown table.** All values are unitless scores, weights, contributions. House style applies.

**Section ordering.** Score breakdown sits **under** the App verdict block, immediately below the `App score:` line and immediately above the `App narrative` subsection. Source: QQQ v2 sample structure:

```
## App verdict: <VERDICT>

**App score:** <##.00> / 100

### Score breakdown
[table]

### App narrative ("Claude's Read")
[narrative]

### Invalidation conditions ("This Trade Is Wrong If")
[from OTA-644]
```

The `App verdict` and `App score:` lines themselves are already rendered by OTA-621's existing v1 assembler. This Story adds the `### Score breakdown` subsection between `App score:` and `### App narrative`.

**Same content for trades and positions.** Both export types render the breakdown identically.
