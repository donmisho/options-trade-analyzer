## Scope
For each scoring criterion (12 TBD + 4 implemented), implement a pure registered formula in `app/options_rules/screening/`, registered via the OTA-726 mechanism under the exact name in the SHARED `formula_registry` contract. Each reads `named_values`, applies `params` (no literals), returns a float in `[0,100]`.

## Acceptance criteria
- All 16 scoring criteria are implemented as registered formulas; each name matches a row in the SHARED `formula_registry` set (OTA-689); OTA-699's membership check passes with no drift.
- Each formula is pure, takes `(named_values, params)`, returns `[0,100]`; no threshold/weight literals — all come from `params`.
- Each formula's math matches its `business-rules.md` catalog definition; for the 4 previously-implemented criteria, behavior matches the legacy implementation (the legacy code is then dead — flag it for removal, do not leave a second live path).
- Where a criterion depends on a producer that is a documented audit fix (e.g. true IV percentile from OTA-720), the formula consumes the corrected named value, and any intentional difference from legacy output is noted.

## Out of scope
Post-scoring adjustments (cushion → OTA-728, asymmetry → OTA-729), gate-phase formulas (earnings → OTA-730, NegativeEV → OTA-731), inline `evaluation_routes` gates (superseded — OTA-780, not in this feature), engine-internal weight cleanup (OTA-733), junction weight/threshold values (those are config, OTA-680).
