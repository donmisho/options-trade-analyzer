## Relevant Context — Do Not Deviate Without Escalation

**Source: insight_engine.md §6.3 (via OTA-700, Code & Test Complete)**
The engine's expression library resolves `formula:<name>` by looking the name up in the **registered rule library** and calling the implementation with the candidate's named values and the rule's bound parameters: `(named_values, params)`. OTA-700 implements the **lookup**; this story supplies the **registry** the lookup resolves against.

**Source: OTA-689 (SHARED formula_registry contract, Code & Test Complete) + OTA-695 OD-1 (option c)**
The named-formula-registry contract is the SHARED `engine_lookups` set `('SHARED', 'formula_registry')` — one row per `formula:<name>` referenced by seeded rules. **Every name registered by this library must match a row in that set.**

**Source: OTA-699 (startup validation, Code & Test Complete) — dual validation, OD-1 (c)**
OTA-699 validates `formula_ref` at load against **two** sources, with **distinct** errors:
1. membership in the SHARED `formula_registry` lookup set (the contract — "is this name in the contract"),
2. implementation in this live registered rule library (the registry — "does an implementation exist").
Both must reject the load independently. **Drift** — a name present in only one source — is itself a third distinct error. This mechanism must expose the live registry in the exact shape OTA-699's check consumes.

**Source: CLAUDE.md — SKILL.md / no hardcoded values; purity**
Formulas are pure functions `(named_values, params) -> float in [0,100]`; thresholds/weights come from junction rows, never literals. Rule-library functions must be pure (no shared state) per the engine's forward concurrency commitment.

## Scope
Implement the formula registration mechanism (decorator / registry hook) for `app/options_rules/screening/`: a module registers an implementation under a name; the engine's `formula:<name>` lookup (OTA-700) resolves it; OTA-699's live-registry membership check reads it. This is the live-registry side of the OD-1 dual-validation contract.

## Acceptance criteria
- A formula module can register an implementation under a name, and the engine resolves `formula:<name>` to it via OTA-700's lookup.
- The live registry is exposed in the shape OTA-699's formula-membership check consumes; OTA-699's live-registry check passes for every registered name.
- Contract/registry drift (a name in only one of the SHARED `formula_registry` set vs. this live registry) is surfaced through OTA-699's drift error path — not silently tolerated.
- The mechanism imposes the purity contract: registered callables take `(named_values, params)` and return a float in `[0,100]`; no shared mutable state.
- No formula implementations yet (those are OTA-727–731) — only the mechanism, validated with a trivial stub formula in tests.
