Options Analyzer

FastAPI + React options analysis, portfolio tracking, and AI-powered trade evaluation platform

📋
Executive Summary
High-level overview of the system architecture

Options Analyzer is a full-stack options analysis and portfolio tracking platform built on FastAPI (Python) and React (Vite). It provides multi-strategy scoring, AI-powered trade evaluation via Azure Foundry (Claude), real-time market data from Charles Schwab, and automated position monitoring with health grading.

The architecture follows a provider-adapter pattern for all external integrations, making it straightforward to add new data sources, AI models, or brokerage connections. A plugin-based strategy system on both frontend and backend allows new trading strategies to be added with zero changes to existing code.

Infrastructure runs on Azure (App Service + Static Web Apps + SQL + Key Vault + Entra ID) with CI/CD through GitHub Actions featuring a dev-first deployment gate and one-click rollback.

Key Design Principles

Provider Adapter Pattern — All external sources implement abstract interfaces. Zero engine/route changes to add a provider.

Strategy Plugin System — Frontend configs + backend scorers. New strategy = 2 files.

SKILL.md Prompts — AI prompts live in Markdown files, never hardcoded in Python.

BFF Auth — Entra ID OIDC + encrypted HTTP-only session cookies. No tokens in browser JS.

Hard Gates — Deterministic quality gates (earnings, negative EV) run before AI evaluation.

Architecture Insights
Notable patterns, decisions, and trade-offs

Provider Adapter Pattern — Zero-Touch Extensibility

All external sources (market data, AI, signals) implement abstract base classes. The ProviderRegistry manages instantiation and lifecycle. Adding a new brokerage or AI model means writing one adapter class — zero changes to routes, engines, or frontend. This pattern has already proven itself: Schwab replaced Tradier as primary provider with no business logic changes.

BFF Authentication — No Tokens in Browser

The Backend-For-Frontend pattern with Entra ID means the React SPA never touches JWT tokens or secrets. The SessionManager encrypts session data into HTTP-only cookies. CSRF protection via double-submit cookie pattern. Certificate-based client assertion (entra-bff-cert from Key Vault) handles the OIDC token exchange server-side.

SKILL.md — Prompts as Configuration

AI prompts are never hardcoded in Python. They live as Markdown files under app/skills/ with variable placeholders. The SkillLoader fills templates at runtime. This allows prompt iteration without code changes and keeps the AI contract visible and version-controlled.

Hard Gates — Deterministic Before Probabilistic

Before any AI evaluation runs, deterministic hard gates check for disqualifying conditions (earnings window, negative expected value). Gates are registered at startup in evaluation order (first-match-wins). This saves AI costs and prevents nonsensical evaluations. New gates can be added by implementing the gate interface and registering at startup.

Lifespan Initialization — Explicit Dependency Wiring

FastAPI's lifespan context manager wires all dependencies in a specific order: DB → Secrets → Auth → Providers → Schwab → AI → Gates → Telemetry → Scheduler → MCP. Each route module exposes an init_*() function that receives its dependencies. No global mutable state, no import-time side effects. Shutdown follows reverse order with graceful resource cleanup.

Deploy Pipeline — Dev-First Social Contract

CI/CD uses a 4-stage model: build-on-push → deploy-to-dev → smoke test → deploy-to-prod. The dev-first gate is a social contract (not automated) — a deliberate scoping choice at solo-dev scale. One-click rollback via artifact re-deploy provides ~3-5 minute RTO.
